# 深入理解实验 — 从 Socket 到线程池

> 不只是"跑通代码"，而是理解每一行为什么这么写。

---

## 一、TCP Socket 到底做了什么

### 1.1 现实类比

想象你开了一家外卖店：

| 现实 | 对应 API |
|------|----------|
| 装一部电话 | `socket()` |
| 去电信局登记号码 | `bind()` |
| 开启来电等待 | `listen()` |
| 接起电话 | `accept()` |
| 顾客拨号 | `connect()` |
| 通话 | `send()` / `recv()` |
| 挂电话 | `close()` |

顾客和店员之间有一条独享的通话线路——这就是 **TCP 连接**。

### 1.2 两个套接字的本质区别

很多人最困惑的地方：

```c
int socket_fd    = socket(...);   // 监听套接字
int connected_fd = accept(...);   // 已连接套接字
```

**socket_fd（监听套接字）**：
- 服务器启动时创建一次，一直存在
- 专门"接电话"，不传数据
- 类比：公司总机号码

**connected_fd（已连接套接字）**：
- 每次 accept 成功生成一个
- 专门跟某个客户端传数据
- 用完即关
- 类比：通话建立后分配的分机线

这就是为什么父进程要关 connected_fd（不负责跟客户聊天），子进程要关 socket_fd（不负责接新电话）。如果弄反了，要么内存泄漏，要么功能错乱。

### 1.3 TCP 三次握手发生在哪里

```
客户端 connect() ───SYN───→  服务端 listen() 队列
客户端         ←──SYN+ACK──  服务端
客户端 connect() ───ACK───→  服务端 accept() 返回 connected_fd
```

- `listen(fd, 10)` 的 10 是内核中等待握手的队列长度
- 握手完成后，连接从队列移到 accept 就绪状态
- `accept()` 只是从就绪队列取走一个连接

## 二、为什么 SO_REUSEADDR 这么重要

### 2.1 TCP 四次挥手

```
主动关闭方                          被动关闭方
    ────FIN────→
    ←───ACK────
    ←───FIN────
    ────ACK────→  （进入 TIME_WAIT，等 2MSL ≈ 60秒）
```

主动关闭方会进入 TIME_WAIT 状态 60 秒。这 60 秒内端口被占用，`bind()` 会失败 `Address already in use`。

### 2.2 SO_REUSEADDR 做了什么

```c
int opt = 1;
setsockopt(socket_fd, SOL_SOCKET, SO_REUSEADDR, &opt, sizeof(opt));
```

允许你在 TIME_WAIT 期间重新 bind 同一个端口。内核说："好吧，虽然旧连接还在等，但新连接可以复用这个端口。"

**不添加这行的话：**每次 Ctrl+C 停掉服务器再启动都要等 60 秒，开发体验极差。

## 三、多进程并发 — fork 的陷阱

### 3.1 fork 到底复制了什么

```c
int pid = fork();
```

fork 之后，子进程是父进程的**副本**。注意这个"副本"包含：

| 复制的内容 | 后果 |
|-----------|------|
| 代码段 | 共享（只读，无需复制）|
| 数据段 | 写时复制（COW），修改时才真正复制 |
| **文件描述符表** | **完整复制！** |
| 堆、栈 | 写时复制 |

**重点：文件描述符表被复制了。** 这意味着 fork 之后，父进程和子进程持有**相同编号的 fd**，指向**同一个内核文件对象**。

### 3.2 为什么一定要关 fd

fork 后的状态：

```
父进程: socket_fd=3, connected_fd=4    ← 两个都在
子进程: socket_fd=3, connected_fd=4    ← 两个都在（复制来的）
```

子进程如果**不关 socket_fd=3**：
- 内核认为监听套接字还有进程在使用
- 引用计数不会归零
- 父进程想关监听端口？关不掉，因为子进程还持有引用

父进程如果**不关 connected_fd=4**：
- 每次都 accept 产生一个新的 connected_fd
- 父进程持有越来越多无用的 fd
- 最终耗尽进程的文件描述符上限（默认 1024）
- 再也没法 accept 新连接

正确做法：

```c
if (pid == 0) {
    close(socket_fd);      // 子：我不接电话，我不管监听
    // ... 处理客户端 ...
    close(connected_fd);   // 子：处理完，挂断
    exit(0);
} else {
    close(connected_fd);   // 父：这个客户已经交给子进程了，我不需要
    // 继续循环 accept
}
```

### 3.3 僵尸进程问题

子进程 exit 后，如果父进程不回收，它会变成**僵尸进程**（Z 状态）。僵尸进程不占 CPU 和内存，但占用一个 PID（进程号）。

解决：

```c
// 方式1：信号处理
signal(SIGCHLD, sigchld_handler);

void sigchld_handler(int sig) {
    while (waitpid(-1, NULL, WNOHANG) > 0);
}

// 方式2：在父进程循环中用 WNOHANG 非阻塞回收
waitpid(-1, NULL, WNOHANG);
```

## 四、多线程并发 — 竞态条件

### 4.1 线程与进程的根本区别

| | 进程 | 线程 |
|---|------|------|
| 地址空间 | 独立 | **共享** |
| 文件描述符 | 独立 | **共享** |
| 全局变量 | 独立 | **共享** |
| 创建开销 | 大（复制页表）| 小（只创建栈和TCB）|
| 一个崩溃 | 不影响其他 | **可能全崩** |
| 通信方式 | IPC（管道/共享内存）| 直接读写共享变量 |

线程共享地址空间 —— 这是它的最大优势（通信快）也是最大风险（数据竞争）。

### 4.2 经典的 connected_fd 竞争问题

**错误的写法：**

```c
int connected_fd;
while (1) {
    connected_fd = accept(socket_fd, NULL, NULL);  // 写入
    pthread_create(&t, NULL, handler, &connected_fd);  // 传递栈地址
}
```

问题在哪？时间线：

```
时刻1: 主线程 accept → connected_fd = 5
时刻2: 主线程创建线程A，传 &connected_fd
时刻3: 主线程下一轮循环 accept → connected_fd = 6（覆盖！）
时刻4: 线程A 读取 *arg → 得到 6（错误！应该是 5）
```

线程 A 读到的可能是 6 而不是 5，因为主线程已经把栈变量覆盖了。

**正确的写法：**

```c
int *connected_fdp = malloc(sizeof(int));    // 堆上分配
*connected_fdp = accept(socket_fd, NULL, NULL);
pthread_create(&t, NULL, handler, connected_fdp);
// 线程内部：free(connected_fdp); 用完释放
```

每个线程有自己独立的堆内存，互不干扰。

### 4.3 pthread_detach 的作用

```c
pthread_detach(client_thread);
```

线程结束后自动回收资源，不需要主线程去 `pthread_join`。适合"每连接一线程"模式，因为主线程不需要等子线程结束。

## 五、线程池 — 为什么比"来一个创一个"好

### 5.1 问题分析

实验二的 `server_thread.c` 每来一个客户端就创建一个线程。问题：

1. **创建/销毁开销**：`pthread_create` 需要分配 TCB、栈（默认 8MB）、初始化。1000 个客户端 = 1000 次创建销毁。
2. **资源不可控**：来 10000 个客户端怎么办？系统会被线程淹没。
3. **栈内存浪费**：1000 个线程 × 8MB 栈 = 8GB 虚拟内存。

### 5.2 池化思想

```
[主线程] ──accept──→ [任务队列] ──取任务──→ [工作线程1]
                       ↑   ↓              [工作线程2]
                      环形缓冲区           [工作线程3]
                                          [工作线程4]
                                          [工作线程5]
```

- 5 个线程预创建好，一直在循环取任务
- 任务队列满了就丢弃（或阻塞等待）
- 线程数固定，资源可控

### 5.3 互斥锁保护什么

```c
void enqueueTask(TaskQueue *q, Task task) {
    pthread_mutex_lock(&task_mutex);        // ← 从这
    q->rear = (q->rear + 1) % q->capacity;
    q->tasks[q->rear] = task;
    q->size++;
    pthread_mutex_unlock(&task_mutex);      // ← 到这都是"临界区"
}
```

为什么需要锁？假设两个线程同时修改 `rear`：

```
线程A: rear = (5 + 1) % 100 = 6
线程B: rear = (5 + 1) % 100 = 6    ← 两个都算出来 6！
线程A: tasks[6] = task_A
线程B: tasks[6] = task_B           ← 覆盖了！
```

加了锁之后，任意时刻只有一个线程能进入临界区，其他线程在外面排队等待。

### 5.4 条件变量 vs 忙轮询

**忙轮询（原版 server_threadpool.c）：**

```c
while (queue->size == 0) {
    usleep(10000);  // 睡 10ms 再试
}
// 取任务
```

问题：CPU 不断醒来检查，即使队列一直是空的，白白消耗 CPU。

**条件变量（server_threadpool_cv.c）：**

```c
pthread_mutex_lock(&mutex);
while (queue->size == 0)
    pthread_cond_wait(&not_empty, &mutex);  // 完全阻塞，零 CPU
// 被 pthread_cond_signal 唤醒后才继续
task = queue->tasks[front];
pthread_mutex_unlock(&mutex);
```

`pthread_cond_wait` 做了三件事：
1. 原子性地释放 mutex（让别人能进入临界区）
2. 阻塞当前线程（CPU 不消耗）
3. 被唤醒后重新获取 mutex

这就实现了"队列空时线程睡觉，有新任务时精准唤醒"。

## 六、表达式计算器原理

### 6.1 递归下降法

我们的 `calculate.c` 使用**递归下降法**，这是最直观的表达式解析方法。

表达式语法（BNF）：

```
expr   → term  ( '+' term | '-' term )*
term   → factor ( '*' factor | '/' factor )*
factor → NUMBER | '(' expr ')'
```

- `expr()` 处理加减：先算 `term()`，然后循环处理 `+` `-`
- `term()` 处理乘除：先算 `factor()`，然后循环处理 `*` `/`
- `factor()` 处理数字或括号

### 6.2 执行过程示例：`2+3*4`

```
expr()
  term()
    factor() → val=2, tok='+'
    (没有 * 或 /，退出 term，返回 2)
  当前 tok='+': next_token
  term()
    factor() → val=3, tok='*'
    当前 tok='*': next_token
    factor() → val=4, tok=END
    v = 3 * 4 = 12
  返回 v=12
  v = 2 + 12 = 14
返回 14
```

### 6.3 为什么乘除优先于加减

关键在于**调用顺序**：

- `expr()` 调 `term()` 再处理 `+-`（加减最后算）
- `term()` 调 `factor()` 再处理 `*/`（乘除中间算）
- `factor()` 取数字（数字优先取）

语法树：

```
     +
    / \
   2   *
      / \
     3   4
```

乘除在语法树的更深处，递归自然会先计算它们。

## 七、关键概念速记

| 概念 | 一句话解释 |
|------|-----------|
| TCP | 可靠的双向字节流，打电话的模型 |
| socket() | 创建通信端点 |
| bind() | 给端点分配地址（IP+端口）|
| listen() | 宣布"我可以接受连接了" |
| accept() | 取走一个等待中的连接 |
| connect() | 主动发起连接 |
| fork() | 克隆当前进程，返回两次 |
| 僵尸进程 | 子进程已死但父进程没回收 |
| SO_REUSEADDR | 允许重用 TIME_WAIT 端口 |
| 竞态条件 | 多线程同时访问共享数据，结果取决于执行顺序 |
| 互斥锁 | 同一时刻只让一个线程进入临界区 |
| 条件变量 | 让线程在某个条件不满足时睡觉 |
| 线程池 | 预创建线程+任务队列，复用线程 |
| 递归下降 | 按语法规则的层次递归解析表达式 |
| htonl/htons | 主机字节序→网络字节序（大端）|
| INADDR_ANY | 0.0.0.0，监听本机所有 IP |
